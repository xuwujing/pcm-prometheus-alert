# 旧项目参考分析

参考项目：

```text
D:\pcm\project\bajcommon-alarm
```

## 1. 可参考设计

1. core + starter + demo 的模块结构
2. `AsyncAlarm` 的异步发送思路
3. `AlertManager` 的告警持续时间、冷却、恢复机制
4. `AlertMessage` 的统一消息封装
5. 基于 Micrometer registry 的指标检查思路
6. Druid 慢 SQL 检测思路
7. starter 自动装配方式

## 2. 不建议沿用的设计

1. 绑定 Dubbo 作为主要采集场景
2. 绑定内部 `bajcommon-metrics-core`
3. 绑定内部小居、飞书工具类
4. 配置类大量使用内部热配置机制
5. 大量逻辑集中在少数大类中
6. demo 模块过多，不利于开源用户理解

## 3. 新项目取舍

新项目保留：

1. 异步推送
2. 告警聚合和冷却
3. 指标阈值判断
4. starter 接入模式

新项目去除：

1. Dubbo 默认依赖
2. 私有内部依赖
3. 单一 IM 平台绑定
4. 重型配置中心假设
5. 本地开发对 Prometheus、SkyWalking、ELK 的强依赖

## 4. 迁移思路

1. 先实现通用告警模型。
2. 再实现 Spring Web 异常采集。
3. 再实现 webhook 推送。
4. 最后按扩展模块实现 SQL、Prometheus、Dubbo 等可选能力。

## 5. 版本取舍

旧项目存在多个 demo 和 starter 模块，新项目主线先选择 Spring Boot 2.x。

Spring Boot 3.x 与 1.x 不进入 MVP，避免一开始引入 `javax.servlet` / `jakarta.servlet`、自动装配文件和依赖版本的兼容复杂度。
